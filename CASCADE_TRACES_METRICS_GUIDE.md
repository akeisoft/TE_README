# 📜 Traces Tab — лог прогонов cascade

## Что это

Каждая строка таблицы = **один вызов** `/cascade/analyze` с persist=true. Это полная история того что делал каскад.

## Структура таблицы

### TIME

Когда запустился прогон. Два формата:

- **"22h ago"**, **"23h ago"** — относительное время для свежих
- **"2026-04-22 12:20"** — абсолютное для более старых

Логика переключения: недавние (≤ 24ч) — relative, дальше — absolute.

### CASCADE

Имя cascade config, который выполнялся. Сейчас везде `default-gemini-diverse` — единственный активный, в будущем планировалось добавить каскад OpenAI\DeepSeek и GROK если это будущее наступит.

Если появится A/B эксперимент (например `experimental-classifier-v2`), здесь будет видно **какой именно каскад** отработал на этом прогоне. Удобно для сравнения.

### STATUS

Одно из трёх значений:

| Значение            | Цвет       | Что означает                                                               |
| ------------------- | ---------- | -------------------------------------------------------------------------- |
| **success**         | 🟢 зелёный | Прогон завершился без ошибок. Результат получен (возможно пустой).         |
| **error**           | 🔴 красный | Один из агентов упал. См. колонку Error.                                   |
| **short_circuited** | —          | Прогон завершился досрочно, но это не ошибка (`has_promise: false` и т.п.) |

**Заметка:** short_circuit обычно показан как `success` со стрелочкой `@step 2`. Это потому что short-circuit — **не ошибка**, это нормальный случай когда classifier решил "обещания нет" и pipeline правильно прервался.

### DURATION

Сколько миллисекунд занял **весь прогон** от начала до конца. Включает:

- Время всех 4 (или меньше при short-circuit) API-вызовов к Gemini
- Время валидатора
- Время записи в БД
- Network overhead

**Интерпретация диапазонов:**

| Время           | Интерпретация                                                               |
| --------------- | --------------------------------------------------------------------------- |
| `7000-10000ms`  | Short-circuit на step 2 (~8 сек) — очень быстро, только vision + classifier |
| `10000-15000ms` | Short-circuit на step 3 — vision + classifier + deadline                    |
| `15000-25000ms` | Полный прогон все 4 шага — **нормально**                                    |
| `25000-35000ms` | Полный прогон, **медленнее среднего** — возможно aggregator тормозил        |
| `> 45000ms`     | **Подозрительно долго** — стоит посмотреть Details                          |

### SHORT-CIRCUIT

Если pipeline оборвался досрочно — тут будет `@step N`:

| Значение          | Что случилось                                               |
| ----------------- | ----------------------------------------------------------- |
| **`—`** (прочерк) | Пробежали все 4 шага, дошли до конца                        |
| **`@step 2`**     | Classifier сказал "обещания нет" → step 3, 4 не запускались |
| **`@step 3`**     | Deadline extractor не нашёл дедлайна → step 4 не запускался |

**Хороший паттерн:** много `@step 2` = каскад эффективно фильтрует мусор.

### ERROR

Если status=error — тут первые 60-80 символов error message. Пример:

```
CascadeEngine: step 4 (aggregator, agent=gemini-aggregator) failed: GeminiCascad...
```

Разбираем по частям:

- **`CascadeEngine`** — источник ошибки (не Elysia, не БД, а сам движок каскада)
- **`step 4 (aggregator, agent=gemini-aggregator)`** — где упало и какой агент был на этом шаге
- **`failed: GeminiCascad...`** — причина (обрезано; см. Details для полного текста)

### ACTION → Details

Кликаешь — открывается modal с **полным step_traces JSON** этого прогона. Главный инструмент отладки. См. [раздел Details Modal](#-details-modal--deep-dive-в-один-прогон) ниже.

## Фильтр "All cascades"

Dropdown сверху справа. Полезен когда у тебя **несколько cascades** (один основной + A/B экспериментальный). Можно отфильтровать только одного.

С одним cascade — dropdown пока бесполезен, но инфраструктура готова (почти, ибо этот ДжаваСкрипт затрахал меня, ублюдский язык, всё перепишем на Ruby on Rails).

# 📊 Metrics Tab — агрегированная статистика

## Что это

**Сводка всех persist-прогонов за выбранное временное окно.** Отвечает на вопросы:

- "Как работает каскад в целом?"
- "Становится ли лучше / хуже?"
- "Где узкое место по скорости?"

Это "дашборд здоровья системы" — для **периодического обзора**, не для отладки инцидентов.

## Фильтр окна (справа)

Dropdown **"Last 24h"** — выбор окна расчёта. Обычно доступны:

- **Last 24h** — свежая оперативка
- **Last 7d** — недельная динамика
- **Last 30d** — месячные тренды

Все метрики ниже считаются **только для прогонов в этом окне**.

## 5 main cards

### 1. TOTAL RUNS

```
5
window: 2026-04-22T16:26 → 2026-04-23T16:26
```

**Сколько persist-прогонов** было в выбранном окне. Плюс точные timestamps начала и конца окна.

**Референс:**

- На staging с тестами: 5-50 runs/day — нормально
- В production с 10 операторами × 30 скринов/день: **300 runs/day** минимум
- Enterprise с 100 операторами: тысячи runs/day

### 2. SUCCESS RATE

```
100.0%
5 successful
```

Доля прогонов со статусом `success` (включая short_circuited) среди всех.

**Как читать:**

| Значение   | Интерпретация                                                    |
| ---------- | ---------------------------------------------------------------- |
| **100%**   | 🟢 Идеально — ни одной ошибки                                    |
| **> 98%**  | 🟢 Норма для healthy pipeline                                    |
| **95-98%** | 🟡 Периодические сбои, терпимо (обычно transient Gemini timeout) |
| **< 95%**  | 🔴 Системная проблема, срочно смотреть Error breakdown           |

### 3. ERROR RATE

```
0.0%
0 errors
```

Обратная Success Rate. Доля прогонов со статусом `error`.

**Когда эта цифра растёт** — срочно в Traces → фильтр по status=error → разобрать первые 3-5 кейсов → понять общую причину.

**Типичные причины ошибок:**

- Gemini API timeout (transient)
- MaxTokens truncation (промпт слишком длинный)
- Invalid JSON из модели
- Rate limit (429 от Gemini)

### 4. SHORT-CIRCUIT RATE

```
60.0%
3 runs exited early
```

**Доля прогонов закончившихся до финала**. 60% значит 3 из 5 прогонов завершились до aggregator'а.

> **Важно:** Short-circuit = **не ошибка**, а оптимизация! Classifier или deadline extractor **правильно решили** что обещания тут нет → сэкономили вызовы aggregator'а.

**Интерпретация:**

| Значение   | Что означает                                                                               |
| ---------- | ------------------------------------------------------------------------------------------ |
| **> 50%**  | 🟢 Каскад **хорошо фильтрует** мусор. Нормально — большинство скринов не содержат обещаний |
| **30-50%** | 🟡 Умеренно. Норма для "рабочих" переписок где много обещаний                              |
| **< 20%**  | 🟠 Подозрительно. Либо все обещания настоящие (редко), либо classifier слишком "лояльный"  |
| **100%**   | 🔴 Все short-circuited → **никогда не создаются обещания**. Что-то сильно сломано          |

**Идеальный диапазон: 40-70%.**

### 5. AVG DURATION

```
18919ms
successful only
```

**Среднее время** по успешным прогонам. Включает и short-circuit (~8 сек), и полные (~22-32 сек).

**"successful only"** — ошибки не учитываются (они могут упасть на разных шагах и исказить среднее). Считаем только нормальные прогоны.

**Как читать:**

| Значение   | Интерпретация                       |
| ---------- | ----------------------------------- |
| **< 15s**  | 🟢 Быстро — оператор не ждёт долго  |
| **15-25s** | 🟢 Нормально                        |
| **25-30s** | 🟡 Терпимо, но можно оптимизировать |
| **> 50s**  | 🟠 Медленно, нужна оптимизация      |

### 6. P50 / P95

```
15347 / 32316 ms
```

**Персентили** — более информативны чем Avg:

| Метрика          | Значение | Что означает                                                       |
| ---------------- | -------- | ------------------------------------------------------------------ |
| **P50 (median)** | 15347 ms | Половина прогонов **быстрее** 15.3 сек. Типичный прогон            |
| **P95**          | 32316 ms | 95% прогонов быстрее 32.3 сек. **Только 5% медленнее этой планки** |

**Зачем нужны персентили:**

- **Avg** чувствителен к выбросам. Один прогон в 60 сек перетянет средний вверх.
- **P50 (median)** показывает "типичный опыт". Если P50 = 15s — большинство юзеров ждут ~15 сек.
- **P95** показывает "худший опыт обычного пользователя". Если P95 = 32s — 5% юзеров ждут дольше полуминуты. Неприятно но терпимо.

**Разрыв P50 vs P95:**

| Ratio P95/P50 | Интерпретация                                          |
| ------------- | ------------------------------------------------------ |
| **< 2.0×**    | 🟢 Стабильная latency. Все прогоны примерно одинаковые |
| **2.0-3.0×**  | 🟡 Умеренная вариативность (твой случай: 2.1×)         |
| **> 3.0×**    | 🟠 Есть медленные прогоны-выбросы. Проверь Traces      |

**Что влияет на P95:**

- Размер PNG (большие скриншоты → медленнее vision_reader)
- Сложность чата (много сообщений → больше токенов)
- Gemini API throttling (когда rate limit близок)
- Thinking models (непредсказуемое thinking time)

## 📊 Avg duration per step role (bar chart)

Главная диагностическая метрика — **где тормозит конкретно**.

```
vision_reader       11124ms  5 calls  ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓
classifier           3192ms  5 calls  ▓▓▓▓▓
deadline_extractor   1367ms  2 calls  ▓
aggregator          10136ms  2 calls  ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓
```

### Как читать

- **Левая цифра** = average ms на этом шаге
- **Правая цифра** = сколько раз этот шаг запускался за окно

### Интерпретация отдельных агентов

#### Vision Reader

- Запускается **каждый раз** (step 1, никогда не пропускается)
- Обычно **самый медленный** (OCR скриншота через Gemini Vision)
- **Норма:** 8-12 сек
- **Оптимизация:** сжимать PNG на клиенте, уменьшать resolution

#### Classifier

- Запускается столько же раз сколько vision (на каждом прогоне)
- **Норма:** 2-4 сек (thinking-модель, но lite)
- Если > 5 сек — возможно thinking level слишком высокий

#### Deadline Extractor

- Запускается **меньше** раз чем classifier (только когда classifier сказал `has_promise: true`)
- **Норма:** 1-3 сек (простая задача — найти фразу)
- Самый быстрый обычно

#### Aggregator

- Запускается только когда deadline extractor нашёл дедлайн
- **Норма:** 3-8 сек (зависит от maxTokens и сложности output)
- Если > 10 сек — возможно maxTokens слишком большой

### Общая формула latency

```
Полный прогон (все 4 шага)  ≈ vision + classifier + deadline + aggregator
                            ≈ 11124 + 3192 + 1367 + 10136
                            ≈ 25819 ms  (25.8 сек)

Short-circuit на step 2     ≈ vision + classifier
                            ≈ 11124 + 3192
                            ≈ 14316 ms  (14.3 сек)

Short-circuit на step 3     ≈ vision + classifier + deadline
                            ≈ 11124 + 3192 + 1367
                            ≈ 15683 ms  (15.7 сек)
```

## 📊 Calls count — распределение short-circuit

Сколько раз **этот agent-role** запускался за окно.

```
vision_reader:      5 calls
classifier:         5 calls
deadline_extractor: 2 calls  ← на 3 меньше!
aggregator:         2 calls  ← тоже на 3 меньше
```

**Разница показывает где short-circuit случился:**

- Step 1 (vision) всегда 5 — запускается первым, никогда не пропускается
- Step 2 (classifier) 5 — запускается на каждом прогоне где был vision
- Step 3 (deadline) **2** — **3 раза short-circuit'нулось** на classifier (5-3=2)
- Step 4 (aggregator) 2 — если дошли до deadline, обычно дойдут и до aggregator

### Расширенный пример

```
vision_reader:      10
classifier:         10
deadline_extractor:  7     ← 3 short-circuit на classifier
aggregator:          4     ← ещё 3 short-circuit на deadline
```

Значит:

- **3 раза** classifier сказал "не обещание" (10 - 7)
- **3 раза** deadline не смог найти дедлайн (7 - 4) — это тоже не ошибка, бывает
- **4 раза** дошли до финала и создали agreement

# 🔍 Details Modal — deep-dive в один прогон

Это **главный инструмент отладки**. Открывается кликом на **Details** кнопку справа в любой строке Traces.

## Структура modal

Content modal обычно разделён на **3 секции**:

### 1. Header — общая информация

```
Trace ID:            abc-123-def-456
Cascade:             default-gemini-diverse
Organization:        <org-uuid>
Operator:            <user-uuid>
Status:              success
Total duration:      22651ms
Short-circuited:     —
Created at:          2026-04-22 12:19:33
```

**Что важно:**

- **Trace ID** — уникальный ID прогона, можешь использовать в SQL-запросах
- **Short-circuited** — на каком шаге оборвался (или `—` если прошёл все)
- **Error message** — появляется только если status=error

### 2. Step Traces — детали каждого агента

Самая важная часть. Массив из 1-4 объектов (по числу запущенных шагов). Для каждого шага:

#### Step header

```
STEP 1: vision_reader
Agent: gemini-vision-reader (gemini-2.5-flash)
Duration: 6200ms
Tokens: in=2070, out=314
Status: ✓ completed
```

#### Input (что передали агенту)

Для vision_reader это **base64 PNG + prompt**:

```json
{
  "image": "<base64 PNG, 800 KB>",
  "messenger": "telegram",
  "systemPrompt": "Ти — VISION READER..."
}
```

Для classifier / deadline / aggregator — **структурированный JSON**:

```json
{
  "contactName": "Петрович",
  "messages": [
    { "direction": "incoming", "text": "Привет", "time": "19:45" },
    {
      "direction": "outgoing",
      "text": "Куплю бухла сегодня в 23:00",
      "time": "19:47"
    }
  ]
}
```

#### Output (что агент вернул)

```json
{
  "has_promise": true,
  "reasoning": "Последнее outgoing содержит обещание '...' от первого лица в будущем времени",
  "promise_draft": {
    "content": "Купити бухло",
    "givenTo": "Петрович",
    "source_text": "Куплю бухла сегодня в 23:00"
  }
}
```

#### Meta-информация

Для каждого шага обычно показано:

- **`ms`** — сколько заняло это конкретно
- **`tokensIn` / `tokensOut`** — input и output токены
- **`model`** — какая модель использовалась
- **`shortCircuitReason`** — если прогон оборвался здесь, почему

### 3. Final output — что попало в БД

```json
{
  "new_agreements": [
    {
      "content": "Купити бухло",
      "givenTo": "Петрович",
      "dueDate": "2026-04-22T20:00:00.000Z",
      ...
    }
  ],
  "toClose": []
}
```

Плюс результат валидатора (сколько agreements отклонил и почему).

## Как использовать Details для отладки

### Сценарий 1: "Почему этот прогон так долго (30+ сек)?"

1. Открой Details → Step Traces
2. Смотри **duration** каждого шага
3. Найди самый долгий → это узкое место
4. Возможные причины по каждому агенту:
   - **vision_reader медленный** → большой PNG, проверь `tokensIn` (если > 3000 — скриншот слишком детальный)
   - **aggregator медленный** → большой output, проверь `tokensOut` (если ~6000 — maxTokens наверняка упёрся в лимит)

### Сценарий 2: "Cascade создал кривое обещание"

1. Details → посмотри output каждого шага
2. Где **первый раз** появилась неправильная информация?
   - Кривое в **vision_reader output** → модель неправильно прочитала скриншот
   - Кривое в **classifier output** → промпт classifier'а неточный
   - Кривое в **deadline_extractor** → промпт deadline неточный
   - Кривое в **aggregator output** → aggregator "творчески" изменил данные

**Правило:** ищи **первое появление ошибки** — там и искать причину.

### Сценарий 3: "Почему short-circuit случился?"

1. Details → последний шаг который отработал
2. Проверь его **output**:
   - Classifier short-circuit → `has_promise: false` + `reasoning` показывает почему модель так решила
   - Deadline short-circuit → `deadline_source: null` + `reasoning` показывает что нет фразы дедлайна

**Если short-circuit кажется неправильным** (должно было создаться обещание) — правильно обновлять промпт того агента который "ошибся".

### Сценарий 4: "Error на step 4"

```
status: error
error_message: "CascadeEngine: step 4 (aggregator, agent=gemini-aggregator) failed:
                GeminiCascadeAdapter: response truncated at maxTokens"
```

1. Details → Step 4 → смотри `input`
2. Смотри сколько `tokensOut` хотел вернуть (возможно уперся в лимит)
3. Решение: увеличить `maxTokens` через Agents tab → Edit

## Полезные SQL-запросы (если UI тормозит)

Иногда быстрее прямо в БД:

```sql
-- Полный step_traces конкретного прогона
SELECT step_traces FROM cascade_trace
WHERE id = '<trace-id>';
-- (вывод в JSON формате — читай через python3 -m json.tool)

-- Самые медленные прогоны за 24h
SELECT id, total_duration_ms, short_circuited_at_step, created_at
FROM cascade_trace
WHERE created_at > NOW() - INTERVAL 24 HOUR
  AND status = 'success'
ORDER BY total_duration_ms DESC LIMIT 10;

-- Все ошибки за 24h
SELECT id, error_message, created_at
FROM cascade_trace
WHERE created_at > NOW() - INTERVAL 24 HOUR
  AND status = 'error'
ORDER BY created_at DESC;
```

# Практические сценарии

## 🔄 Ежедневный

Открыть Metrics → Last 24h → глянуть:

1. **Error rate** — не растёт ли (должен быть < 2%)
2. **Avg duration** — не увеличился ли (дрейф, может Gemini замедлился)
3. **Short-circuit rate** — в norme (40-70%)

Если всё зелёное — закрыл вкладку, работаешь дальше.

## 📅 Еженедельный обзор

**Last 7d** → **сравни с предыдущей неделей**:

- Success rate растёт? Upward trend = работа над промптами двигает к лучшему
- Duration падает? Gemini обновления или твои оптимизации работают
- Short-circuit rate меняется? Сигнал что классификатор стал строже/слабее

## 🚨 Реакция на алерт

Если завтра увидишь **Error rate = 15%**:

1. Metrics → посмотри какой именно агент в per-step чаще всего падает
2. Traces → фильтр по error → первые 5 случаев
3. Details → понять причину (truncation? timeout? invalid JSON?)
4. Fix промпт/config в Agents или Prompts tab

## 🔍 Отладка жалобы от оператора

"Оператор пишет что cascade не создал обещание на его скриншоте":

1. Traces → отсортировано по времени → видишь свежий прогон
2. Status?
   - **success + short-circuit @step 2** → classifier решил "не обещание". Кликни Details → посмотри reasoning. Возможно промпт classifier'а слишком строгий
   - **success без short-circuit** → значит обещание создалось. Проверь `/agreements/current-list` — может просто не обновилась лента UI
   - **error** → читай error message + Details для deep dive

# Ключевые формулы и бенчмарки

## Эталонные значения для healthy cascade

| Метрика            | Идеально | Норма  | Плохо          |
| ------------------ | -------- | ------ | -------------- |
| Success rate       | 100%     | > 98%  | < 95%          |
| Error rate         | 0%       | < 2%   | > 5%           |
| Short-circuit rate | 40-70%   | 30-80% | < 20% или 100% |
| Avg duration       | < 20s    | 15-25s | > 30s          |
| P95 duration       | < 30s    | 25-35s | > 45s          |
| P95 / P50 ratio    | < 2×     | 2-3×   | > 3×           |

## Формула здоровья каскада

```
Cascade Health = SuccessRate - ErrorRate × 10
```

**Пример:**

- `100% success, 0% error` → `100 - 0×10 = 100%` → 🟢 **ИДЕАЛЬНО**
- `95% success, 5% error` → `95 - 5×10 = 45%` → 🟡 **ПРОБЛЕМА**
- `90% success, 10% error` → `90 - 10×10 = -10%` → 🔴 **ХЕРОВО**

Формула не официальная, но хороший heuristic для "сразу увидеть состояние".

## Формула latency budget

**Цель:** весь прогон ≤ 25 сек (для комфортного UX оператора).

Разбивка по агентам:

- Vision Reader: ≤ 12 сек (48%)
- Classifier: ≤ 4 сек (16%)
- Deadline Extractor: ≤ 3 сек (12%)
- Aggregator: ≤ 6 сек (24%)

Если какой-то агент **выходит за свой budget** — это точка для оптимизации.

## Минимально достаточное окно для статистики

- **5-10 runs** — слишком мало, метрики шумные, делать выводы нельзя
- **20-50 runs** — минимум для первых выводов
- **100+ runs** — уверенные выводы
- **500+ runs** — можно делать тонкие оптимизации на основе данных
