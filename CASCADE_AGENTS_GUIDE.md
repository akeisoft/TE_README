<a name="1"></a>
## 1. Что такое Cascade

### Кратко

**Cascade** — это замена монолитного X-RAY pipeline на серию из 4 специализированных AI-агентов, каждый из которых делает свою работу. Вместо одного гигантского промпта "найди все обещания на скриншоте и верни JSON" — 4 коротких промпта, каждый решает одну подзадачу.

### Зачем

**Проблема с X-RAY:**
- Один огромный промпт (~15000 токенов), который делает всё: OCR, классификацию, извлечение дедлайнов, генерацию JSON
- Сложно отлаживать: если обещание не распознано — непонятно где сломалось (плохо прочитал? неправильно классифицировал? не нашёл дедлайн?)
- Промпт приходится править целиком — risky
- Нельзя экспериментировать с разными моделями на разных задачах

**Решение cascade:**
- **Специализация** — каждый агент заточен под одну задачу, короткий промпт (1200-1500 токенов)
- **Прозрачность** — видно на каком шаге ошибка. Traces показывают вход и выход каждого агента
- **Гибкость моделей** — можно использовать дешёвую модель для OCR и мощную для классификации
- **A/B эксперименты** — меняешь один агент, не трогая остальные
- **Precision over recall** — можно настроить строгую валидацию, отсечь false positives

<a name="2"></a>
## 2. Архитектура: pipeline из 4 агентов

### Схема

```
 ┌──────────────────────┐
 │    base64 (PNG)     │
 │ Operator: user@org   │
 └──────────┬───────────┘
            │
            ▼
 ┌─────────────────────────────────────────┐
 │ Agent 1: Vision Reader                  │   Читает код base64 → JSON с сообщениями
 │   Model: gemini-2.5-flash               │   Output: { contactName, messages[] }
 │   Thinking: none                        │
 └──────────┬──────────────────────────────┘
            │ messages[]
            ▼
 ┌─────────────────────────────────────────┐
 │ Agent 2: Classifier                     │   Последнее outgoing → обещание или нет?
 │   Model: gemini-3.1-flash-lite          │   Output: { has_promise, confidence, ... }
 │   Thinking: low                         │
 └──────────┬──────────────────────────────┘
            │
            │ ⚡ SHORT-CIRCUIT: если has_promise=false → пропускаем шаги 3-4
            │
            ▼
 ┌─────────────────────────────────────────┐
 │ Agent 3: Deadline Extractor             │   Находит точную фразу дедлайна
 │   Model: gemini-3-flash-preview         │   Output: { deadline_source: "завтра в 15:00" }
 │   Thinking: medium                      │
 └──────────┬──────────────────────────────┘
            │
            │ ⚡ SHORT-CIRCUIT: если deadline_source=null → пропускаем шаг 4
            │
            ▼
 ┌─────────────────────────────────────────┐
 │ Agent 4: Aggregator                     │   Собирает всё в финальный JSON
 │   Model: gemini-2.5-flash               │   Output: { new_agreements[], toClose[] }
 │   Thinking: none                        │
 └──────────┬──────────────────────────────┘
            │
            ▼
 ┌──────────────────────┐
 │ Validator (code)     │   Проверяет 9 правил (дедлайн есть, content ≥2 слов, ...)
 └──────────┬───────────┘
            │
            ▼
 ┌──────────────────────┐
 │ Timezone Fix (code)  │   naive "YYYY-MM-DD HH:MM" → UTC ISO
 └──────────┬───────────┘
            │
            ▼
 ┌──────────────────────┐
 │ DB + WebSocket push  │   MySQL INSERT + уведомление клиента
 └──────────────────────┘
```

### Почему именно 4 агента

| # | Агент | Роль | Почему отдельный |
|---|---|---|---|
| 1 | Vision Reader | OCR | Нужна визуальная модель (multi-modal). Другим агентам картинка не нужна — только текст |
| 2 | Classifier | Решение "обещание/не обещание" | Нужна мощная модель для семантики. Дешевле запустить её один раз, а не внутри большого промпта |
| 3 | Deadline Extractor | Поиск фразы времени | Узкая задача, где важна дословная точность — нужна модель с thinking |
| 4 | Aggregator | Сборка JSON | Детерминированная задача — дешёвая модель, без thinking |

Преимущество: каждая модель делает то что у неё получается лучше всего. Мощные thinking-модели — только там где нужна логика. Дешёвые — для механики.

<a name="3"></a>
## 3. Агент 1 — Vision Reader

### Роль

Читает код base64 месенджера как картинку и возвращает JSON со списком видимых сообщений. **Ничего не интерпретирует** — только фотографически копирует текст.

### Что на входе

- Код base64 (PNG, ~500-700 KB)
- Messenger context: `telegram` / `whatsapp` / `viber`
- Current time (Kyiv local — для будущих агентов)

### Что на выходе

```json
{
  "contactName": "AKEI SERVICE",
  "messages": [
    { "direction": "incoming", "text": "Привет!", "time": "14:30" },
    { "direction": "outgoing", "text": "Привезу завтра в 15:00", "time": "14:31" }
  ]
}
```

### Рекомендуемые настройки

| Параметр | Значение | Почему |
|---|---|---|
| **Model** | `gemini-2.5-flash` | Поддерживает vision, дешевле чем 3.x. 3.x Flash тоже подойдёт, но дороже |
| **Temperature** | `0.1` | Низкая — нам нужна точная копия текста, не творчество |
| **Max Tokens** | `4096` | Достаточно для ~12 сообщений. Если чат плотный — можно 6144 |
| **Thinking Level** | `none` (или `minimal` для 3.x) | OCR не требует рассуждений. Thinking здесь — трата денег |

### Симптомы неправильных настроек

| Проблема | Вероятная причина | Решение |
|---|---|---|
| Сообщения пропущены | Низкий maxTokens, oбрезано | Поднять до 6144 |
| Неточно скопирован текст (опечатки) | Слишком высокая temperature | Снизить до 0.0-0.1 |
| Сообщения "додуманы" (галлюны) | Слишком высокая temperature | Снизить до 0.0 |
| Неправильно определены направления | Скриншот низкого качества или модель устарела | Проверить картинку, попробовать 3.x Flash |

<a name="4"></a>
## 4. Агент 2 — Promise Classifier

### Роль

Смотрит на **последнее исходящее сообщение** (последний bubble справа) и решает: содержит ли оно настоящее обещание?

### Что считается обещанием

✅ **Да:**
- 1-е лицо будущего времени: "зроблю", "куплю", "передзвоню"
- + явный дедлайн: "завтра", "о 15:00", "до п'ятниці"

❌ **Нет:**
- Приказ другому: "зроби", "купи"
- Уже выполненное: "зробив", "купив"
- Неуверенность: "може", "спробую", "якщо вийде"
- Рефлексивное: "зробиться", "вирішиться"
- Без дедлайна: "зроблю" (когда — неизвестно)

### Что на входе

Output vision_reader (messages[]).

### Что на выходе

```json
{
  "has_promise": true,
  "confidence": 0.95,
  "reasoning": "Outgoing message contains future tense action with explicit deadline",
  "promise_draft": {
    "content": "Привезти товар",
    "givenTo": "AKEI SERVICE"
  }
}
```

### Рекомендуемые настройки

| Параметр | Значение | Почему |
|---|---|---|
| **Model** | `gemini-3.1-flash-lite-preview` | Оптимум цена/качество. Lite справляется с классификацией, полный 3 Flash избыточен |
| **Temperature** | `0.0` | Нужно детерминированное решение — одинаковый вход → одинаковый выход |
| **Max Tokens** | `2048` | Ответ короткий (has_promise + reasoning) |
| **Thinking Level** | `low` | Это задача на рассуждения — thinking помогает отличить "куплю завтра" от "может куплю". **Не убирай thinking** — упадёт качество ахереть как сильно |

<a name="5"></a>
## 5. Агент 3 — Deadline Extractor

### Роль

Находит в тексте последнего outgoing-сообщения **точную фразу-дедлайн** — дословную подстроку. 

### Примеры

| Текст сообщения | deadline_source |
|---|---|
| "Приеду завтра в 15:00" | `"завтра в 15:00"` |
| "Куплю через годину" | `"через годину"` |
| "До п'ятниці зроблю" | `"до п'ятниці"` |
| "Зроблю скоро" | `null` (размытое) |
| "Зроблю" | `null` (без времени) |

### Что на входе

Output vision_reader + output classifier.

### Что на выходе

```json
{
  "deadline_source": "завтра в 15:00",
  "confidence": 0.98
}
```

Или, если дедлайна нет:
```json
{
  "deadline_source": null,
  "confidence": 0.9
}
```

### Рекомендуемые настройки

| Параметр | Значение | Почему |
|---|---|---|
| **Model** | `gemini-3-flash-preview` | Мощнее классификатора — должен понимать нюансы "скоро" vs "завтра" |
| **Temperature** | `0.0` | Нужна дословная цитата, а не парафраз |
| **Max Tokens** | `5120` | Текущее значение (у тебя) — с запасом. Достаточно 2048, но больше не повредит |
| **Thinking Level** | `medium` | Важно подумать: это явный дедлайн или размытая фраза? Medium даёт лучшее качество |

<a name="6"></a>
## 6. Агент 4 — Aggregator

### Роль

Собирает выходы трёх предыдущих агентов в финальный JSON который ожидает сервер (`analyzeV2Response`).

### Логика

```
IF classifier.has_promise == false OR deadline_extractor.deadline_source == null:
    new_agreements: []
ELSE:
    new_agreements: [{
        content: из classifier.promise_draft.content,
        givenTo: из classifier.promise_draft.givenTo,
        dueDate: из deadline_source (raw text, сервер нормализует),
        openedAt: current time,
        source: из контекста,
        excuse: 1 предложение объяснения
    }]

toClose: []
contactName, messages: из vision_reader
```

Это **детерминированная сборка** — никаких решений не принимается, просто копирование полей.

### Что на входе

Outputs всех 3 предыдущих агентов.

### Что на выходе

```json
{
  "contactName": "AKEI SERVICE",
  "messages": [ /* все из vision_reader */ ],
  "new_agreements": [
    {
      "content": "Привезти товар",
      "givenTo": "AKEI SERVICE",
      "dueDate": "завтра в 15:00",
      "openedAt": "2026-04-22 15:30",
      "source": "telegram",
      "excuse": "Оператор зобов'язався привезти товар завтра о 15:00"
    }
  ],
  "toClose": []
}
```

### Рекомендуемые настройки

| Параметр | Значение | Почему |
|---|---|---|
| **Model** | `gemini-2.5-flash` | Задача механическая — дешёвая модель ок |
| **Temperature** | `0.1` | Небольшая гибкость для поля `excuse` (читаемое объяснение) |
| **Max Tokens** | **`6144`** | **Важно!** Output содержит все messages из vision_reader. На плотном чате (12+ bubbles × ~50 токенов) → 600+ токенов только messages, плюс JSON overhead. 2048 недостаточно! |
| **Thinking Level** | `none` | Рассуждать не надо — просто копирование полей |

### Симптомы неправильных настроек

| Проблема | Вероятная причина | Решение |
|---|---|---|
| **HTTP 500 "truncated at maxTokens"** | **maxTokens слишком низкий (обычно 2048)** | **Поднять до 6144** |
| Обещания не создаются, хотя все agents OK | Aggregator игнорирует промпт | Проверить промпт, попробовать другую модель |
| `excuse` поле неправильное / неудобочитаемое | Промпт или temperature | Чуть поднять temp до 0.2-0.3 |
| Потеряны messages в output | maxTokens → усечение | Поднять maxTokens |

<a name="7"></a>
## 7. Поля формы "Edit agent"

Когда ты нажимаешь `Edit` напротив агента в Cascade Monitor → Agents tab, видишь форму. Разберём каждое поле.

### NAME

Внутреннее имя агента. Используется в seed.ts и в traces. **Не меняй** — если изменишь, сломаются cascade configs которые на него ссылаются.

Допустимые значения: `gemini-vision-reader`, `gemini-promise-classifier`, `gemini-deadline-extractor`, `gemini-aggregator`.

### DESCRIPTION

Человекочитаемое описание для UI. Можно менять свободно — это только комментарий.

### PROVIDER

LLM-провайдер. Сейчас поддерживается только `gemini`. В будущем можно добавить `openai`, `anthropic`, `local-ollama` и т.д. (нужно написать adapter).

### MODEL

Название модели провайдера. **Сейчас в твоей установке используются:**

- `gemini-2.5-flash` — быстрая и дешёвая, **поддерживает vision**, без thinking (vision_reader, aggregator)
- `gemini-3-flash-preview` — новое поколение Flash, **с thinking** (deadline_extractor)
- `gemini-3.1-flash-lite-preview` — **самая дешёвая** thinking-модель (classifier)

**Правило:** сложная задача → thinking-модель (3.x). Простая механика → 2.5 Flash.

**⚠️ Поле это text input — сервер не валидирует**. Опечатка "gemini-2.5-flesh" пройдёт в сохранение, но упадёт при первом вызове Gemini API. Будь внимателен.

### TEMPERATURE

Диапазон **0.0 - 1.0**. Контролирует "креативность":
- `0.0` — детерминированно. Одинаковый вход → одинаковый выход. Для классификации, извлечения фактов, OCR.
- `0.1 - 0.3` — лёгкая вариативность. Для генерации текста (типа поля `excuse`).
- `0.5+` — не используй в production.

### MAX TOKENS

Максимум токенов в **output** агента. Если агент попытается вернуть больше — Gemini обрежет ответ посередине, JSON будет невалидный → HTTP 500 "truncated at maxTokens".

Правило: **output_budget ≥ ожидаемый_размер_ответа × 2** (запас на "дыхание").

Типичные значения по ролям:
- OCR (vision_reader) — 4096-6144
- Classifier — 1024-2048
- Deadline extractor — 1024-5120
- Aggregator — **6144+** (содержит все messages!)

### THINKING LEVEL

Только для Gemini 3.x моделей! Для 2.5 игнорируется.

- `none` (или пусто) — без рассуждений, быстро
- `minimal` — короткое планирование, для Flash
- `low` — умеренные рассуждения (~500 токенов внутренне)
- `medium` — глубокие рассуждения (~1500 токенов)
- `high` — максимум (~3000+ токенов), медленно и дорого

**Важно:** thinking токены **отдельно тарифицируются** и замедляют ответ на 3-15 секунд. Не включай если не нужно.

Где использовать thinking:
- Classifier — `low` (различать обещание vs не-обещание)
- Deadline extractor — `medium` (отличать "завтра" от "скоро")
- Aggregator — **никогда** (чистая механика)
- Vision reader — **никогда** (OCR, не рассуждения)

### API KEY

Опциональное поле. Если пусто — используется `GEMINI_API_KEY` из environment. Если заполнено — этот агент использует свой ключ (полезно для изоляции rate limits).

**На production оставляй пусто** — ключ в `.env` на сервере.

### SYSTEM PROMPT

Инструкция для агента. Длинная строка на украинском/английском. Определяет поведение агента. Меняется реже всего — если что-то работает, не трогай.

Если меняешь — **не удаляй ключевые разделы** (ПРАВИЛА, ФОРМАТ ВЫХОДА, ВАЖНО). Агент начнёт галлюцинировать.

<a name="8"></a>
## 8. Thinking Level — что это и зачем

### Что такое thinking

Gemini 3.x (и новее) поддерживает "thinking mode": перед ответом модель генерирует **внутренние рассуждения**, которые не возвращаются пользователю, но улучшают качество финального ответа.


### Уровни

| Level | Токенов на thinking | Задержка | Когда использовать |
|---|---|---|---|
| `none` | 0 | — | Простые задачи: OCR, механика |
| `minimal` | ~100-300 | +1-2с | Лёгкое планирование, Flash only |
| `low` | ~300-600 | +3-5с | Классификация, принятие простых решений |
| `medium` | ~1000-2000 | +5-10с | Извлечение данных с нюансами |
| `high` | ~2500-5000 | +10-20с | Сложные задачи: математика, код, многоэтапное рассуждение |

### Когда thinking помогает, а когда нет

✅ **Помогает:**
- Классификация "обещание / не обещание" — нужно рассуждение о семантике
- Отличить "скоро" от "завтра в 15:00"
- Понять контекст разговора

❌ **Не помогает (только тратит деньги/время):**
- OCR — просто чтение картинки или кода base64
- Копирование полей из одного JSON в другой
- Форматирование вывода

### Какие модели поддерживают

В текущей установке используются 3 модели:

| Модель | Поддерживает thinking? | Уровни | Где используется |
|---|---|---|---|
| `gemini-2.5-flash` | ❌ | — | vision_reader, aggregator |
| `gemini-3-flash-preview` | ✅ | все 4 уровня | deadline_extractor |
| `gemini-3.1-flash-lite-preview` | ✅ | все 4 уровня | classifier |

**Если выберешь модель без thinking support** (например, `gemini-2.5-flash`) и выставишь thinking-level — сервер проигнорирует его и запрос пройдёт без thinking. В логе появится warning.

**⚠️ UI не валидирует модель** — поле `Model` это text input. Ты можешь вписать что угодно, сервер передаст строку в Gemini API. Если модель не существует — получишь ошибку от Gemini при ближайшем cascade-прогоне. Проверяй орфографию.

<a name="9"></a>
## 9. Short-circuit rules (короткие замыкания)

### Что это

Правило "если шаг вернул определённый результат — пропустить остаток pipeline". Экономит время и деньги.

### Где применяется

**Шаг 2 → Шаг 3+4 (skip):**
- Правило: `classifier.has_promise === false`
- Логика: если нет обещания, нет смысла искать дедлайн и агрегировать

**Шаг 3 → Шаг 4 (skip):**
- Правило: `deadline_extractor.deadline_source === null`
- Логика: если фраза-дедлайн не найдена, нет смысла запускать aggregator

### Что происходит при short-circuit

1. Pipeline останавливается
2. В traces записывается `shortCircuitedHere=true` на соответствующем шаге
3. `new_agreements: []` в финальном output
4. HTTP 200, но agreement не создаётся

### Где настраивается

В `seed.ts` (код) для каждого cascade step. Через UI **не меняется** — это логика pipeline. Если хочешь добавить/убрать — коммити в код.

Пример из seed.ts:
```ts
{
  agentName: "gemini-promise-classifier",
  stepOrder: 2,
  role: "classifier",
  shortCircuitRule: { field: "has_promise", equals: false },
  isFinal: false,
}
```
