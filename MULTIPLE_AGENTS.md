# Зачем в Cascade разные агенты, а не один универсальный

**Версия:** 1.0 · **Дата:** 04.04.2026 · **Проект:** TalkExplorer Cascade

---

## WTF?

Почему не использовать одну версию Gemini 🧐

Разные агенты/модели дают **несколько независимых точек зрения** на одну задачу. Это даёт устойчивость к ошибкам одной модели, повышает точность там где модели "спорят", и позволяет подобрать правильный инструмент под каждую задачу.

> **Аналогия:** вместо одного врача-универсала — консилиум из 3 специалистов. Каждый смотрит со своей стороны, итог получается более надёжный.

---

## 1. Разнообразие = устойчивость к системным ошибкам

### Проблема одной модели

У любой LLM есть **свои слепые зоны** — категории входов где она регулярно ошибается:

- **Gemini 2.5-flash** иногда путает украинские инфинитивы ("зробити") с приказами
- **Gemini 3.x Flash** иногда "додумывает" дедлайны которых нет в тексте
- **Gemini 3.1-flash-lite** быстрая, но слабее в длинном контексте (иногда очень тупая)

Если все 4 шага делает одна модель — все 4 ошибки будут **одинакового типа**. Один и тот же баг проявится на 4 уровнях сразу.

### Как разные модели это решают

У нас сейчас:

| Шаг | Модель | Роль модели |
|---|---|---|
| Vision | `gemini-2.5-flash` | OCR, простая задача |
| Classifier | `gemini-3.1-flash-lite` | Быстрая семантика |
| Deadline | `gemini-3-flash-preview` | Точное извлечение фраз |
| Aggregator | `gemini-2.5-flash` | Детерминированная сборка |

Когда одна модель ошибается на каком-то edge-case — другая часто ловит ошибку. **Ансамбль стабильнее чем единичная модель** — это классический вывод в ML.

### Реальный пример

У нас была огромная проблема с инфинитивами ! Gemini_2.5 иногда говорил "это не обещание" на фразы типа "зробити документи на завтра". Если бы **та же модель** делала следующий шаг, она бы тоже не нашла дедлайн. А так — другая модель 3.0 (deadline extractor) **может** найти его независимо.

## 2. Инструмент под задачу (model-task fit)

Разные модели **оптимизированы под разное**. 

### Наша раскладка и обоснование

#### Vision Reader → `gemini-2.5-flash`

- **Задача:** base64\OCR (чтение картинки → текст)
- **Что нужно:** поддержка vision, скорость, дешевизна
- **Почему не 3.x:** для OCR thinking не нужен, 3.x дороже без выгоды
- **Результат:** дёшево и быстро 

#### Classifier → `gemini-3.1-flash-lite`

- **Задача:** "обещание или приказ?" — **семантическое** решение
- **Что нужно:** понимание нюансов языка, **thinking** для разбора двусмысленности
- **Почему не 2.5:** 2.5 не поддерживает thinking, на двусмысленных кейсах путается
- **Почему lite:** полный 3 Flash избыточен для бинарной классификации
- **Результат:** умнее чем 2.5 на нюансах, но недорого

#### Deadline Extractor → `gemini-3-flash-preview`

- **Задача:** найти **точную фразу** времени в тексте
- **Что нужно:** дословная точность, thinking чтобы отличить "скоро" от "завтра в 15:00"
- **Почему не lite:** lite слабее на задаче "выдели подстроку по критерию"
- **Почему не pro:** pro дорогой и медленный, для этой задачи хватает Flash
- **Результат:** точное извлечение без галлюнов

#### Aggregator → `gemini-2.5-flash`

- **Задача:** **детерминированная сборка** JSON из уже готовых частей
- **Что нужно:** стабильность, скорость, дешевизна
- **Почему не 3.x:** thinking тут **вреден** — модель начнёт "переосмысливать" и менять данные классификатора
- **Результат:** быстро и механически собирает ответ

### Если бы все были на одной модели

**Везде `gemini-3-pro`:**
- Vision: оверкилл, 5× цена за OCR
- Classifier: хорошо (но так же как 3.1-lite)
- Deadline: хорошо (но так же как 3-flash)
- Aggregator: **начнёт модифицировать данные** из-за излишнего thinking — хуже чем 2.5

**Везде `gemini-2.5-flash`:**
- Vision: отлично
- Classifier: **хуже на нюансах** (нет thinking)
- Deadline: **хуже на дифференциации** ("скоро" vs "завтра")
- Aggregator: отлично

> **Смешанный подход — оптимум по качеству и по цене.**

## 3. Single responsibility = короткие промпты

Отдельный агент = отдельный простой промпт.

### Сравнение

#### Монолитный промпт (X-RAY):

```
Ты анализируешь скриншот мессенджера.
1. Прочитай все сообщения
2. Определи какие из них обещания (учитывая...)
   - 1-е лицо будущее vs приказы
   - исключения для инфинитивов
   - blacklist слов
3. Если обещание — найди в тексте дедлайн
4. Нормализуй deadline в формат YYYY-MM-DD HH:MM
5. Проверь не дубликат ли
6. Собери JSON строго такого формата: { ... 200 строк схемы ... }
7. Учитывай что иногда Telegram показывает preview
8. Если whatsapp — игнорируй таймер disappearing messages
...
(всего ~15000 токенов инструкций)
```

Модель **теряется** в такой длине. Забывает ранние правила когда доходит до поздних. Путает приоритеты.

#### Cascade промпты:

```
Vision Reader: "Ты OCR. Читай скриншот. Возвращай JSON."      (1300 токенов)
Classifier:    "Ты решаешь обещание/не обещание.              (1500 токенов)
                Смотри на последнее outgoing."
Deadline:      "Ты извлекаешь фразу дедлайна. Дословно."      (1400 токенов)
Aggregator:    "Ты собираешь JSON из 3 входов."               (1300 токенов)
```

Каждый промпт — **как в фокусе**. Модель делает одно дело хорошо.

## 4. Контрольные точки для валидации

### Разные агенты = разные точки для контроля

Между шагами можно:
- Проверять output структурно (Zod schema validation)
- Отсекать явно невалидные результаты
- Запускать short-circuit
- Кэшировать промежуточные результаты


---

## 5. Когда лучше один агент

Сascade - Минусы:

❌ **Медленнее** — 4 последовательных вызова ~15с vs 1 вызов ~5с

❌ **Сложнее** — больше moving parts, сложнее отлаживать

❌ **Избыточен для простых задач** — Гораздо сложнее в настройке, я уже затрахался эти промпты настраивать !
